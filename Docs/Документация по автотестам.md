# Умный OCR - Автотесты

## Тест-кейс: Проверка создания проекта (Распознавания документов в утверждённой форме)

**Файл:** `smart-ocr.json`

### Назначение

Проверка создания проекта для распознавания документов в утверждённой форме (Smart OCR).

### Предусловия

- Пользователь авторизован с правами администратора
- Базовый URL системы доступен
- Учётные данные корректны

### Шаги выполнения

| # | Шаг | Метод | Endpoint | Описание |
|---|-----|-------|----------|----------|
| 1 | Авто-логин | POST | `/auth/account` | Получение JWT токена для авторизации |
| 2 | Создание проекта | POST | `/api/Projects` | Создание нового проекта OCR |
| 3 | Получение проекта | GET | `/api/Projects/{id}` | Проверка созданного проекта |
| 4 | Проверка схемы разметки | GET | `/api/MarkingSchemes/project/{id}` | Проверка автоматического создания схемы |
| 5 | Удаление проекта | DELETE | `/api/Projects/{id}` | Очистка тестовых данных |

### Входные данные

#### Создание проекта (POST /api/Projects)

```json
{
  "name": "autotest_ocr_{{timestamp}}",
  "description": "Тестовый проект OCR для автотестов",
  "component": 0,
  "modelUsage": 0
}
```

| Параметр | Тип | Описание |
|----------|-----|----------|
| `name` | string | Уникальное имя проекта (генерируется автоматически) |
| `description` | string | Описание проекта |
| `component` | number | Тип компонента: `0` = Умный OCR |
| `modelUsage` | number | Использование модели: `0` = По умолчанию |

### Ожидаемые результаты

#### Шаг 1: Авторизация
- **Status:** `200 OK`
- **Response:** JWT токен сохранён в environment

#### Шаг 2: Создание проекта
- **Status:** `201 Created`
- **Response:** ID созданного проекта (число)

#### Шаг 3: Получение проекта
- **Status:** `200 OK`
- **Проверки:**
  - `id` совпадает с созданным
  - `name` совпадает с отправленным
  - `description` совпадает с отправленным
  - `component` = `0`
  - `smartOCRModelUsage` = `0`
  - `dataSetCount` = `0`
  - `freeDoc` = `false`

#### Шаг 4: Проверка схемы разметки
- **Status:** `200 OK`
- **Проверки:**
  - Возвращён массив схем разметки
  - Есть минимум одна схема
  - `projectId` схемы совпадает с ID проекта
  - Схема имеет имя

#### Шаг 5: Удаление проекта (TEARDOWN)
- **Status:** `204 No Content` или `200 OK`
- **Действие:** Удаление созданного проекта и очистка переменных

## Тест-кейс: Проверить навигацию по вкладкам проекта OCR

**Файл:** `smart-ocr.json`

### Назначение

Проверка доступности всех вкладок проекта OCR (Датасет, Обучение, Использование, Тестирование).

### Предусловия

- Пользователь авторизован с правами администратора
- Базовый URL системы доступен

### Шаги выполнения

| # | Вкладка | Метод | Endpoint | Описание |
|---|---------|-------|----------|----------|
| 1 | - | POST | `/api/Projects` | Создание тестового проекта OCR |
| 2 | - | GET | `/api/Projects/{id}` | Проверка созданного проекта |
| 3 | Датасет | GET | `/api/MarkingSchemes/project/{id}?parentless=true` | Схемы разметки |
| 4 | Датасет | GET | `/api/ProjectDataImgs/project/{id}?pageNumber=0&pageSize=20&dataSetType=Train` | Изображения датасета |
| 5 | Обучение | GET | `/api/ModelTemplates?component=SmartOCR` | Шаблоны моделей |
| 6 | Обучение | GET | `/api/TrainProcessTemplates/project/{id}` | Шаблоны обучения |
| 7 | Обучение | GET | `/api/Projects/{id}/DataSet/Balance` | Баланс датасета |
| 8 | Использование | GET | `/api/InferenceProcessTemplates/project/{id}` | Шаблоны инференса |
| 9 | Использование | GET | `/api/ModelTypes?component=SmartOCR&SmartOCRModelUsage=0` | Типы моделей |
| 10 | Использование | GET | `/api/models/project/{id}` | Модели проекта |
| 11 | Тестирование | GET | `/api/ModelTypes?activeProjectId={id}` | Типы моделей для тестирования |
| 12 | TEARDOWN | DELETE | `/api/Projects/{id}` | Удаление тестового проекта |

### Входные данные

#### Создание проекта (POST /api/Projects)

```json
{

  "description": "Тест навигации по вкладкам OCR",
  "component": 0,
  "modelUsage": 0
}
```

### Ожидаемые результаты

#### Шаг 1: Создание проекта
- **Status:** `201 Created`
- **Response:** ID созданного проекта

#### Шаг 2: Проверка проекта
- **Status:** `200 OK`
- **Проверка:** `component` = `0` (OCR)

#### Шаги 3-4: Вкладка "Датасет"

- **MarkingSchemes:** Возвращает объект с массивом `result`

- **ModelTemplates:** Возвращает объект с шаблонами моделей
- **TrainProcessTemplates:** Возвращает валидный объект

- **Status:** `200 OK` для всех запросов
- **InferenceProcessTemplates:** Валидный ответ
- **ModelTypes:** Возвращает типы моделей для SmartOCR
- **models/project:** Возвращает модели проекта

#### Шаг 11: Вкладка "Тестирование"
- **Status:** `200 OK`
- **ModelTypes:** Возвращает типы моделей для тестирования

#### Шаг 12: Удаление проекта (TEARDOWN)
- **Status:** `204 No Content` или `200 OK`
- **Действие:** Удаление проекта и очистка переменных

| `name` | string | Уникальное имя проекта |
| `description` | string | Описание проекта |
| `component` | number | Тип компонента: `1` = AI Текст (NLP) |

### Ожидаемые результаты

#### Шаг 1: Создание проекта
- **Status:** `201 Created`
- **Response:** ID созданного проекта

#### Шаг 2: Проверка создания
- **Status:** `200 OK`
- **Проверки:**
  - `name` совпадает с отправленным
  - `description` совпадает с отправленным
  - `component` = `1` (NLP)

#### Шаг 3: Редактирование проекта
- **Status:** `200 OK`

#### Шаг 4: Проверка редактирования
- **Status:** `200 OK`
- **Проверки:**
  - `name` обновлён
  - `description` обновлён
  - `component` = `1` (не изменился)

#### Шаг 5: Удаление проекта (TEARDOWN)
- **Status:** `204 No Content` или `200 OK`

## Тест-кейс: Проверить навигацию по вкладкам проекта NLP

**Файл:** `smart-ai.json`

### Назначение

Проверка доступности всех вкладок проекта AI Текст (Модели, Навыки, Использование, Тестирование).

### Предусловия

- Пользователь авторизован с правами администратора
- Базовый URL системы доступен

### Шаги выполнения

| # | Вкладка | Метод | Endpoint | Описание |
|---|---------|-------|----------|----------|
| 1 | - | POST | `/api/Projects` | Создание тестового проекта NLP |
| 2 | - | GET | `/api/Projects/{id}` | Проверка созданного проекта |
| 3 | Модели | GET | `/api/enums/NlpEngines` | Справочник NLP движков |
| 4 | Модели | GET | `/api/ModelTemplates?component=NLP&allowedForProjectId={id}` | Шаблоны моделей |
| 5 | Навыки | GET | `/api/NlpTask/project/{id}` | Задачи NLP |
| 6 | Навыки | GET | `/api/NlpContextFiles?IsPredefined=true` | Контекстные файлы |
| 7 | Навыки | GET | `/api/NlpSystemMessages?IsPredefined=true` | Системные сообщения |
| 8 | Использование | GET | `/api/InferenceProcesses/agents/llm?projectId={id}` | Процессы инференса |
| 9 | Тестирование | GET | `/api/ModelTypes?component=NLP&ActiveProjectId={id}` | Типы моделей |
| 10 | TEARDOWN | DELETE | `/api/Projects/{id}` | Удаление тестового проекта |

### Входные данные

#### Создание проекта (POST /api/Projects)

```json
{
  "name": "NLP_Nav_Test_{{timestamp}}",
  "description": "Тест навигации по вкладкам NLP",
  "component": 1
}
```

### Ожидаемые результаты

#### Шаг 1: Создание проекта
- **Status:** `201 Created`
- **Response:** ID созданного проекта

#### Шаг 2: Проверка проекта
- **Status:** `200 OK`
- **Проверка:** `component` = `1` (NLP)

#### Шаг 3: Вкладка "Модели" - NlpEngines
- **Status:** `200 OK`
- Возвращает справочник NLP движков

#### Шаг 4: Вкладка "Модели" - ModelTemplates
- **Status:** `200 OK`
- Возвращает объект с шаблонами моделей для NLP

#### Шаг 5: Вкладка "Навыки" - NlpTask
- **Status:** `200 OK`
- Возвращает объект с задачами NLP

#### Шаг 6: Вкладка "Навыки" - NlpContextFiles
- **Status:** `200 OK`
- Возвращает контекстные файлы

#### Шаг 7: Вкладка "Навыки" - NlpSystemMessages
- **Status:** `200 OK`
- Возвращает системные сообщения

#### Шаг 8: Вкладка "Использование" - InferenceProcesses
- **Status:** `200 OK`
- Возвращает процессы инференса для LLM

#### Шаг 9: Вкладка "Тестирование" - ModelTypes
- **Status:** `200 OK`
- Возвращает типы моделей для NLP

#### Шаг 10: Удаление проекта (TEARDOWN)
- **Status:** `204 No Content` или `200 OK`
- **Действие:** Удаление проекта и очистка переменных

## Принципы тестирования

### Автономность

- Каждый тест полностью независим
- Имена проектов генерируются с уникальным суффиксом через `Date.now()`
- Авторизация происходит автоматически перед каждым запуском

### Self-cleanup

- После выполнения теста созданный проект удаляется
- Переменные очищаются в TEARDOWN
- Удаляется только созданный проект (по `createdProjectId`)

### Безопасность данных

- Тест НЕ удаляет существующие проекты
- Тест создаёт только новые проекты с уникальными именами
- После завершения все созданные данные очищаются

## Возможные ошибки

| Ошибка | Причина | Решение |
|--------|---------|---------|
| `401 Unauthorized` | Неверные учётные данные | Проверить `username`/`password` |
| `Connection refused` | Недоступен сервер | Проверить `baseUrl` |
| `500 Internal Server Error` | Ошибка сервера | Проверить логи сервера |
| `Project already exists` | Конфликт имён | Тест использует уникальные имена |
